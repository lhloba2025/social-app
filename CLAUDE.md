# تعليمات العمل على نظام هوفيرا

> هذا الملف محفوظ في كل مستودع من مستودعات هوفيرا الأربعة ليتذكّره Claude تلقائياً في كل جلسة.
> المستخدم هو عبدالله (صاحب النظام، غير مبرمج) — اشرح دائماً بعربي بسيط وواضح.

## هيكل المشروع
المشروع أربعة مستودعات منفصلة على GitHub، تعيش جنباً إلى جنب في نفس المجلد الأب، وكل واحد مستودع مستقل:

- `front-end/` — لوحة التحكم التي يراها المستخدم
  React + Vite + Tailwind، تُنشر على Vercel
- `backend/` — واجهة REST API التي يستدعيها الفرونت لجلب البيانات
  Node/Express + Prisma متعدد المستأجرين (master / tenant / staff)، تُنشر على EC2
- `mobile-app/` — تطبيق الجوال
  Expo / React Native
- `Project Social App/` — تطبيق التسويق
  يُنشر على Railway، مستودعه اسمه social-app تحت حساب lhloba2025

الفرونت ليس لديه منطق بيانات خاص به. كل بيانات ديناميكية في الفرونت مصدرها endpoint في مستودع `backend/`.

## قواعد إلزامية لكل تعديل

1. عند طلب تغيير أو إضافة في الفرونت، افحص أولاً هل يحتاج بيانات أو سلوكاً لا توفره الـ API حالياً (حقل جديد، endpoint جديد، تغيير response، باراميتر جديد، إلخ).
2. إذا كان يحتاج: **لا تزيّف البيانات ولا تكتبها hardcoded في الفرونت.** افتح مستودع `backend/` وسوّ التعديل المطابق في الـ API، وحافظ على تطابق شكل request/response بين المستودعين.
3. نفّذ تعديل الفرونت في `front-end/` وتعديل الباك في `backend/` بحيث يعملان معاً end-to-end.
4. عند الانتهاء، بلّغ المستخدم بوضوح وبشكل منفصل: أي ملفات تغيّرت في كل مستودع.
5. إذا كان التعديل يمس الفرونت فقط (تنسيق، تخطيط، نصوص...) ولا يحتاج شيئاً جديداً من الـ API، قلها صراحةً واترك مستودع الباك بدون أي لمس.

## قاعدة الرفع (مهمة جداً)

- الرفع إلى فرع main ينشر تلقائياً على الإنتاج، وعليه عملاء حقيقيون.
- **لا تنفّذ `git push` أبداً بنفسك.** عبدالله يرفع بنفسه من GitHub Desktop بعد مراجعة الشغل والتأكد منه.
- لا ترفع شيئاً غير مُتحقق منه. تحقّق من صياغة الكود قبل أن تقول إنك خلّصت.
- لا تفترض أبداً أن المستخدم رفع مستودعاً لأنه رفع الآخر — كل مستودع يحتاج commit مستقل في GitHub Desktop.
- ذكّر المستخدم دائماً برفع كل المستودعات التي تعدّلت، وأعطه الخطوات بالأزرار خطوة خطوة.

### الصيغة المطلوبة حرفياً عند انتهاء تعديل فرونت + باك

> "Frontend changes are done. In GitHub Desktop, select the front-end repository, review the changes, and commit + push to the main branch.
> Backend changes are also done. In GitHub Desktop, switch the repository dropdown (top-left) to the backend repository, review those changes, and commit + push to the main branch as well."

## أسلوب العمل

- اشتغل بدون أسئلة كثيرة: اتخذ القرارات التقنية المنطقية بنفسك ونفّذ.
- لا تسأل المستخدم إلا إذا كان القرار يغيّر شيئاً يراه العميل، أو فيه مخاطرة على الإنتاج.

## تفضيل لغوي إلزامي

- لا تخلط كلمة عربية وكلمة إنجليزية في نفس السطر.
- أي مصطلح إنجليزي يوضع في سطر منفصل لوحده.

## ملاحظات معروفة

- ملف `myApiClient.js` في الفرونت يتجاهل باراميتر الترتيب إذا مُرّر كنص مثل
  `list("-created_date")`
  فالترتيب لا يصل للباك، ويجب أن يتم الترتيب في الفرونت (client-side).
- صفحة المحاسبة وأداة توليد الصور نُقلتا من تطبيق التسويق إلى هوفيرا نقلاً أصيلاً.
- الخطة الجارية: نقل بقية أدوات التسويق إلى هوفيرا بالترتيب —
  بطاقات التهنئة، ثم منشئ التصاميم والمكتبة، ثم منظومة النشر — ثم إطفاء Railway.
